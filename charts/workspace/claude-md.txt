# Workspace Environment

This is a Kubernetes-based development workspace running in an isolated container.

## System

- **OS:** Ubuntu 24.04 (container, not a full VM)
- **User:** `dev` (uid/gid 1000), passwordless `sudo` available
- **Shell:** bash
- **Display:** Virtual X11 on `:99` (1280x720) — set `export DISPLAY=:99` for GUI apps

## Storage

- **`/home/dev`** — persistent volume (default 50Gi). Survives pod restarts. All project work belongs here.
- **`/tmp`** — ephemeral. Lost on restart. Do not store anything important here.
- **`~/.ssh`** — symlinked to `/home/dev/.credentials/.ssh` (persistent)
- **`~/.gitconfig`** — symlinked to `/home/dev/.credentials/.config/git` (persistent)
- **`~/.config/gh`** — symlinked to `/home/dev/.credentials/.config/gh` (persistent)
- **Per-tool logins persist too** — `~/.expo`, `~/.docker`, `~/.aws`, `~/.config/gcloud`, `~/.npmrc`, and `~/.git-credentials` are symlinked into `/home/dev/.credentials/`, so `eas login`, `npm login`, `docker login`, `aws configure`, and `gcloud auth` survive pod restarts (no re-login every session). For non-GitHub HTTPS git, opt in per host with `git config --global credential.https://<host>.helper store` (keeps the App-token helper for github.com untouched); creds save to the now-persistent `~/.git-credentials`.

## Claude Task API

HTTP API on port 6080 for remotely running Claude Code tasks in persistent tmux sessions. Supports two auth methods: OAuth2 (browser) or Bearer token (programmatic).

### Get your API token

From inside the pod:
```bash
cat ~/.claude-tasks/.api-token
```

Or via the browser (OAuth2 session):
```bash
curl localhost:6080/api/claude/auth/token
```

### Endpoints

```bash
# Create a task
curl -X POST localhost:6080/api/claude/tasks \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"prompt": "clone repo user/repo", "workdir": "/home/dev"}'

# List all tasks
curl -H "Authorization: Bearer $TOKEN" localhost:6080/api/claude/tasks

# Get task detail + recent output
curl -H "Authorization: Bearer $TOKEN" localhost:6080/api/claude/tasks/{id}

# Get raw output (last N lines)
curl -H "Authorization: Bearer $TOKEN" localhost:6080/api/claude/tasks/{id}/output?tail=100

# Send follow-up prompt to existing task
curl -X POST localhost:6080/api/claude/tasks/{id}/message \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"prompt": "now create a README"}'

# Kill a running task
curl -X DELETE -H "Authorization: Bearer $TOKEN" localhost:6080/api/claude/tasks/{id}

# Regenerate API token (OAuth2 only)
curl -X POST localhost:6080/api/claude/auth/token/regenerate
```

Tasks run in tmux sessions (`claude-{task_id}`). Attach with `tmux attach -t claude-{task_id}` to watch live. Task state persists in `~/.claude-tasks/`.

## Key Constraints

- **Resource limits** — typically 2-3 CPU cores, 3-5Gi RAM. Check usage with `curl localhost:6080/metrics`.
- **Ephemeral /tmp** — anything outside `/home/dev` is lost on pod restart.
- **Package persistence** — `sudo apt-get install` works but packages are lost on restart. For persistent tools, install to `~/.local/bin` and add it to `PATH` in `~/.bashrc`.
- **Network** — full internet access available. Outbound connections work normally.

## Tips

- Use `tmux` for long-running processes to avoid losing work if a terminal session drops.
- **Copying text from the web terminal:** the terminal runs tmux with `mouse on`, so a plain click-drag is captured by tmux (it selects into tmux's own buffer, not your clipboard). Hold **Shift** while dragging to select — that bypasses mouse reporting and gives the browser a native selection, which **⌘/Ctrl+C** then copies to your OS clipboard.
- The VS Code server runs on port 8080 (no auth — protected by ingress-level authentication).
- SSH keys persist in `~/.ssh`. Generate with: `ssh-keygen -t ed25519`
- To check what's running: `curl localhost:6080/health`

### Contributing changes back (fork + PR)

To propose a change to kube-coder itself (not your user-code in this
workspace), use the fork+PR flow. From inside `/home/dev/kube-coder`:

```bash
# One-time: create a fork under YOUR GitHub account and add as a remote
# named `fork`. `origin` stays pointed at the upstream so `git pull`
# fetches the canonical main branch.
gh repo fork --remote --remote-name fork

# Per-change:
git checkout -b descriptive-branch-name
# … make edits …
git commit -am "fix: short description"
git push fork HEAD
gh pr create --base main --repo imran31415/kube-coder
```

Authentication note: `gh repo fork` and `gh pr create` use whatever
identity `gh auth status` reports. By default that's the workspace's
GitHub App token (commits attributed to the bot account). To push as
yourself instead, either run `gh auth login` once (token persists on
the PVC at `~/.config/gh/`) or use SSH agent forwarding from your
local machine when SSH'd in.

**Do not `git push origin`** — `origin` is the upstream repo and the
workspace's identity doesn't have write access there. Use `fork`.

## Persistent Memory (MCP) — check FIRST, never guess

This workspace ships a persistent memory store backed by SQLite at
`/home/dev/.claude-memory/memory.db` and exposed to you via MCP tools.
Entries survive across tasks, browser tabs, and pod restarts, and are
shared with the dashboard's Memory tab — the user can view and edit them.

### Default behavior: ALWAYS check memory before saying "I don't know"

A `<workspace_memories>` block is automatically prepended to every user
prompt with the top-K relevant entries (via a `UserPromptSubmit` hook).
**Read that block before composing your reply.**

If the user asks anything about themselves, their preferences, this
project, or anything they may have previously mentioned — and the
auto-injected block does not contain the answer — **call `memory_search`
BEFORE responding with "I don't know" / "I don't have that information"
/ "you haven't told me"**. Treat memory as authoritative; never claim
ignorance without checking.

Common triggers requiring an explicit `memory_search` call:
- "what is my name / email / role / language / editor / …"
- "what do I prefer for X"
- "do you remember when / what about / did I tell you …"
- "what's our convention for X" / "how do we deploy this project"

Only say "I don't know" *after* the auto-injection block and a
deliberate `memory_search` both come up empty. Then ask the user — and
when they answer, immediately call `memory_remember` so the next session
has it.

### Writing memory

- "remember X" / "note that" / "save this" → call `memory_remember`.
  Pick a stable `namespace.key` (e.g. `user.preferences.editor`,
  `project.kube-coder.deploy_cmd`).
- Observed a stable preference the user did not explicitly ask to
  save? Call `memory_remember` with `importance=0.7`. Keep values
  concise (one fact per entry).
- "forget X" → confirm with the user, then call `memory_forget`.

### Namespaces

- `user.*` — personal facts (name, email, preferences, working style).
- `project.<repo>.*` — facts about a specific project.
- Avoid `session.*` (reserved for future auto-pruning).

### Tagging

Tag sensitive entries with `secret` to opt them out of the
auto-injection block. They remain readable via explicit tool calls but
won't leak into every prompt.

## Agent Orchestration (MCP) — spawn sub-agents for parallel work

This workspace ships an `agent-orchestrator` MCP server that lets you
spawn, monitor, and collect results from sub-agent sessions. Use it to
decompose large problems into parallel subtasks. The same tools are
registered for **Claude, Ante, OpenCode, and Codex**, so any of these
harnesses can spawn any other and lineage is tracked across them.

By default agents spawn in **headless** mode: the sub-agent runs the
prompt to completion and exits, so `get_agent_status` flips to
`completed` (or `error`, with an `exit_code`) on its own and
`wait_for_agent` returns the real result. Pass `mode="interactive"` to
instead launch a long-lived REPL you can attach to from the dashboard
(needed for `kc-harness`, which has no headless mode).

Guardrails: spawning is capped by nesting depth (`KC_MAX_SPAWN_DEPTH`,
default 3) and by concurrent live agents (`KC_MAX_SUBAGENTS`, default 8)
so orchestration can't fork-bomb the pod or run away on API spend.

### Available tools

| Tool | Purpose |
|------|---------|
| `spawn_agent(prompt, assistant, mode)` | Launch a sub-agent in its own tmux session. Returns `task_id` immediately. `mode` is `headless` (default) or `interactive`. |
| `get_agent_status(task_id)` | Check status (`running`/`completed`/`error`/`killed`) and `exit_code`. |
| `list_subagents()` | List all agents you've spawned (auto-detects your task context). |
| `get_agent_output(task_id, tail)` | Retrieve the output log from an agent. |
| `wait_for_agent(task_id, timeout)` | Block until the agent finishes, then return its output. |
| `kill_agent(task_id)` | Terminate a running agent. |

### Supported assistants

- `ante` — Ante CLI (pre-installed; defaults to OpenRouter when its key is set)
- `claude` — Claude Code (requires Anthropic API key or OAuth)
- `codex` — Codex (OpenAI, pre-installed). ChatGPT OAuth — run `codex login`
  ONCE in a terminal to sign in (creds persist on the PVC under `~/.codex`);
  after that it works as a Hypervisor driver, a Build-tab REPL, and a
  `spawn_agent(assistant="codex")` sub-agent, same as Claude. No API key.
- `librefang` — LibreFang agent OS (pre-installed; talks to its registry
  `coder` agent and reuses whatever provider key is in the environment)
- `opencode-openrouter` — OpenCode via OpenRouter
- `opencode-deepseek` — OpenCode via DeepSeek
- `kc-harness` — Opensource GPU (requires a running Ollama endpoint)

### Orchestration patterns

**Parallel fan-out** — split work across many agents at once:
```
spawn_agent(prompt="Refactor module A", assistant="ante")  → task_1
spawn_agent(prompt="Refactor module B", assistant="ante")  → task_2
spawn_agent(prompt="Write tests for module C", assistant="claude") → task_3
...then poll or wait for each...
get_agent_output(task_1)
get_agent_output(task_2)
get_agent_output(task_3)
...synthesize results...
```

**Sequential pipeline** — chain agents where each step depends on the prior:
```
wait_for_agent(spawn_agent(prompt="Step 1: design", assistant="ante").task_id)
wait_for_agent(spawn_agent(prompt="Step 2: implement", assistant="claude").task_id)
```

**Collect-and-synthesize** — spawn workers, gather results, report:
```
agents = [spawn_agent(prompt=f"Analyze file {f}") for f in files]
results = [get_agent_output(a.task_id) for a in agents]
...summarize across all results...
```
